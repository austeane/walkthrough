# Walkthrough Template Reference

Reference for agents that need to evaluate, modify, or generate data consumed by `assets/walkthrough-template.html`.

**Template file**: `/Users/austin/dev/walkthrough/assets/walkthrough-template.html` (~1670 lines)
**Renderer**: `/Users/austin/dev/walkthrough/scripts/render_html.py`

---

## 1. Template Engine

**Engine**: Jinja2 (via `jinja2.Environment` with `autoescape=False`)

### Variables Injected by `render_html.py`

| Variable | Type | Description |
|----------|------|-------------|
| `data` | dict | The full walkthrough JSON structure (after `prepare_data()` processing) |
| `data_json` | string | `json.dumps(data)` — embedded verbatim into `<script>const DATA = {{ data_json }};</script>` for client-side JS |
| `pygments_css` | string | CSS rules generated by `HtmlFormatter(style="monokai", cssclass="highlight", nobackground=True).get_style_defs(".highlight")`. Injected directly into a `<style>` block |

### Rendering Pipeline

```
walkthrough.json
    |
    v
render_html.py --input X --output Y [--normalized Z]
    |
    +--> resolve_media()       # If --normalized provided: resolves screenshot data_uri from JSONL
    +--> prepare_data()        # Deep-copies data, runs highlight_diff_hunk() on each diff_hunk
    |       |
    |       +--> highlight_diff_hunk()  # Adds `hunk["html"]` field with Pygments-highlighted diff
    |
    +--> get_pygments_css()    # Generates Monokai-themed CSS for .highlight class
    +--> template.render(data=data, data_json=data_json, pygments_css=pygments_css)
```

## 2. Data Contract

Every `{{ }}` and `{% %}` expression in the template is documented below. Optional fields are marked; if absent, the template either skips the section (`{% if ... %}`) or uses a `| default()` filter.

### 2.1 Top-Level: `data`

```
data
├── meta
│   └── repo_root          # string, optional — enables editor links
├── overview
│   ├── goal               # string — page <title>, header, overview heading
│   ├── summary            # list[str], optional — bullet points on overview slide
│   ├── key_files          # list[str], optional — file chips on overview slide
│   └── diagram_mermaid    # string, optional — Mermaid diagram source
└── steps                  # list[Step] — the walkthrough step objects
```

### 2.2 `data.meta`

| Field | Used In | Purpose |
|-------|---------|---------|
| `meta.repo_root` | Jinja2 template (key_files links, files_changed links) + JS (`getRepoRoot()`, `absPath()`, `editorLinks()`) | Prepended to relative file paths to build `cursor://file/...` and `vscode://file/...` URIs. Trailing slashes are stripped by JS. |

### 2.3 `data.overview`

| Field | Type | Required | Template Usage |
|-------|------|----------|----------------|
| `overview.goal` | string | Yes (has `default("")`) | `<title>` tag, `.header-title` text, `<h1>` in overview section |
| `overview.summary` | list[str] | No | Rendered as `<ul class="overview-summary">` bullet list. Guarded by `{% if data.overview.summary %}` |
| `overview.key_files` | list[str] | No | Rendered as `<a class="key-file">` chips with `cursor://file/` links. Guarded by `{% if data.overview.key_files %}` |
| `overview.diagram_mermaid` | string | No | Raw Mermaid code placed in `<pre class="mermaid">`. Triggers Mermaid CDN load. Guarded by `{% if data.overview.diagram_mermaid %}` |

**Key file link construction** (Jinja2):
```jinja2
cursor://file/{{ (data.meta.repo_root ~ '/' ~ f) if data.meta.repo_root is defined and data.meta.repo_root else f }}
```

### 2.4 `data.steps[]` (Step Object)

| Field | Type | Required | Template Usage |
|-------|------|----------|----------------|
| `step.id` | string | Yes | Used as `<section id="{{ step.id }}">`, `href="#{{ step.id }}"` in TOC, `data-step-id` on buttons, and as key for `document.getElementById('media-' + step.id)` |
| `step.title` | string | Yes | TOC link text, `<h2>` heading, modal title prefix |
| `step.intent` | string | No | Italic subheading below title. Guarded by `{% if step.intent %}`. Hidden on viewports < 500px tall |
| `step.claims` | list[Claim] | No (defaults to `[]`) | Rendered as `<div class="claim ...">` elements inside `.claims` container |
| `step.evidence` | object | No | Presence enables "View Evidence" button. Contains diff_hunks, commands, media, files_changed |
| `step.decisions` | list[Decision] | No | Rendered in evidence modal (JS only, not Jinja2) |
| `step.errors_encountered` | list[Error] | No | Rendered in evidence modal (JS only, not Jinja2) |

### 2.5 `step.claims[]` (Claim Object)

| Field | Type | Required | Template Usage |
|-------|------|----------|----------------|
| `claim.text` | string | Yes | Text content of the claim `<div>` |
| `claim.confidence` | string | No (defaults to `"grounded"`) | Applied as CSS class on `.claim` element |

**Confidence rendering**:

| Value | CSS Class | Visual Effect |
|-------|-----------|---------------|
| `"grounded"` | `.claim` (no modifier) | Normal text |
| `"inferred"` | `.claim.inferred` | Left border `--badge-inferred` (amber), indented, secondary text color |
| `"speculative"` | `.claim.speculative` | Left border `--badge-speculative` (red), indented, secondary text color, prepended "inferred" badge pill |

Note: The CSS class for `speculative` renders a `::before` pseudo-element with content `"inferred"` (not "speculative"), which appears as a small red badge.

### 2.6 `step.evidence` (Evidence Object)

| Field | Type | Template Usage |
|-------|------|----------------|
| `evidence.files_changed` | list[str] | Step footer: `<a class="meta-tag">` chips with `cursor://file/` links. Modal: listed in "Files Changed" section |
| `evidence.diff_hunks` | list[DiffHunk] | Modal: rendered as `.diff-block` elements |
| `evidence.commands` | list[Command] | Step footer: preview as `<span class="meta-tag">$ cmd | truncate(30)`. Modal: rendered as `.command-block` elements |
| `evidence.media` | list[MediaItem] | Step body: triggers `.media-gallery` in-slide (thumbnails). Modal: rendered in "Screenshots" section |

### 2.7 `evidence.diff_hunks[]` (DiffHunk Object)

| Field | Type | Required | Usage |
|-------|------|----------|-------|
| `diff_hunk.file` | string | Yes | Displayed in `.diff-header`, used for editor links |
| `diff_hunk.line` | number | No (defaults to `1`) | Line number for editor links (`vscode://file/path:line`, `cursor://file/path:line`) |
| `diff_hunk.html` | string | No (added by `render_html.py`) | Pre-rendered Pygments HTML. If present, inserted directly. If absent, JS falls back to `renderDiffLines(before, after)` |
| `diff_hunk.before` | string | No | JS fallback: rendered as `- ` prefixed lines with `.diff-line-del` class |
| `diff_hunk.after` | string | No | JS fallback: rendered as `+ ` prefixed lines with `.diff-line-add` class |
| `diff_hunk.diff` | string | No | Used by `render_html.py` preprocessing: if present and `before`/`after` are absent, this raw diff text is highlighted directly by Pygments |

**Preprocessing by `render_html.py`**: For each hunk, `highlight_diff_hunk()` generates `hunk["html"]` by:
1. If `hunk["diff"]` exists, highlight it with `DiffLexer`
2. Else if `hunk["before"]` or `hunk["after"]` exist, synthesize unified diff lines (`- line` / `+ line`) and highlight with `DiffLexer`
3. Result: `hunk["html"]` = Pygments-formatted HTML wrapped in `.highlight` class

## 3. Layout Structure

The template produces a single-page application with scroll-snap sections. Major structural elements:

```
<html data-theme="dark">
<head>
  <style>
    /* All CSS inline — no external stylesheets except Google Fonts */
    /* Pygments CSS injected via {{ pygments_css }} */
  </style>
</head>
<body>
  .progress-bar              # Fixed top bar, width driven by JS
  nav.sidebar                # Fixed left sidebar (TOC)
    .sidebar-header          # "Walkthrough" title
    ul.toc                   # TOC list
      li.toc-item            # One per step (overview + data.steps)
        a.toc-link[data-step=N]
  header.header              # Fixed top header (right of sidebar)
    .header-left             # Hamburger + title
    .header-right            # Search input + theme toggle
  main.main                  # Scrollable content area
    section.step.overview-step#overview[data-step="0"]
      h1                     # data.overview.goal
      ul.overview-summary    # data.overview.summary[]
      .overview-key-files    # data.overview.key_files[]
      .overview-diagram      # data.overview.diagram_mermaid
    {% for step in data.steps %}
    section.step#{{ step.id }}[data-step="{{ loop.index }}"]
      .step-header           # Step N + title + intent
      .claims                # claim divs
      .media-gallery         # Only if step.evidence.media exists
        .media-thumbs        # Populated by JS from DATA
      .step-footer           # files_changed tags + command tags + "View Evidence" button
    {% endfor %}
  .modal-overlay             # Evidence modal (hidden by default)
    .modal
      .modal-close
      .modal-title
      #modalBody             # Populated by JS: diff_hunks, commands, decisions, errors, media, files_changed
  .media-lightbox            # Image lightbox (hidden by default)
    .media-lightbox-close
    .media-lightbox-prev / .media-lightbox-next
    img#lightboxImg
    .media-lightbox-caption
  <script> (Mermaid CDN, conditional)
  <script>const DATA = {{ data_json }};</script>
  <script> (WalkthroughViewer class)
</body>
```

### Scroll-Snap Behavior

- `html`: `scroll-snap-type: y mandatory`
- Each `.step`: `scroll-snap-align: start`, full viewport height (`100dvh - 48px` for header)
- Slides are designed as full-screen pages, not a continuously scrolling document

## 4. Interactive Features (JavaScript)

All interactivity is in the `WalkthroughViewer` class, instantiated on `DOMContentLoaded`.

### 4.1 Theme Toggle

| Behavior | Details |
|----------|---------|
| Storage | `localStorage.getItem('walkthrough-theme')` |
| Mechanism | Toggles `data-theme` attribute on `<html>` between `"dark"` and `"light"` |
| Icon | Moon (dark) / Sun (light) |

### 4.2 Progress Bar

- Fixed 3px bar at top of viewport
- Width = `(currentStep / (totalSteps - 1)) * 100 %`
- Updated on step change via `IntersectionObserver`

### 4.3 IntersectionObserver

- `threshold: 0.5` — step is "current" when 50% visible
- Adds `.visible` class (triggers reveal animations)
- Updates TOC active state
- Updates progress bar

### 4.4 Keyboard Navigation

| Key | Action |
|-----|--------|
| `ArrowDown`, `ArrowRight`, `Space`, `PageDown` | Next step |
| `ArrowUp`, `ArrowLeft`, `PageUp` | Previous step |
| `Home` | First step |
| `End` | Last step |
| `Escape` | Close modal/sidebar |

Keyboard navigation is disabled when focus is in an `<input>` or when the modal is open (except `Escape`).

### 4.5 Touch/Swipe

- Tracks `touchstart` Y → `touchend` Y on `main`
- Threshold: 50px vertical swipe triggers step change

### 4.6 Search

- Filters steps by `textContent.toLowerCase().includes(query)`
- Hides non-matching steps (`display: none`) and TOC links (`.search-hidden`)
- Real-time on `input` event

### 4.7 Evidence Modal

- Opened by clicking "View Evidence" button (`.evidence-btn[data-step-id]`)
- Closes on: close button click, overlay click, `Escape` key
- Body content is built dynamically by `renderEvidence()` in JS using the embedded `DATA` object
- Sections rendered (in order): Diff Hunks, Commands, Decisions, Errors, Screenshots, Files Changed

### 4.8 Media Galleries

- Initialized by `initMediaGalleries()`: iterates `DATA.steps`, finds containers by `id="media-{step.id}"`, populates with `renderMedia()`
- **Grouping logic**: Items with matching `group` field are grouped. If a group contains items with `group_role: "before"` and `group_role: "after"`, they render in a side-by-side `.media-compare` grid. Otherwise, grouped items render sequentially. Ungrouped items render individually.
- **Thumbnail display**: Uses `thumbnail_uri` if available, falls back to `data_uri`

### 4.9 Lightbox

- Activated by clicking any `.media-thumb` element (delegated event on `document`)
- Collects all `.media-thumb` siblings in the same `.media-thumbs` or `.media-compare` container for gallery navigation
- Navigation: left/right arrow buttons, `ArrowLeft`/`ArrowRight` keys, wraps around
- Shows full-size image (`data-full-src` attribute, which maps to `media.data_uri`)
- Displays caption text below image

### 4.10 Sidebar (Mobile)

- Hidden off-screen on viewports <= 768px (`transform: translateX(-100%)`)
- Toggled by hamburger button (`.hamburger`)
- Closed when TOC link clicked or `Escape` pressed

### 4.11 Mermaid Diagrams

- CDN loaded conditionally: `{% if data.overview.diagram_mermaid %}`
- URL: `https://cdn.jsdelivr.net/npm/mermaid@10/dist/mermaid.min.js`
- Initialized with theme matching current `data-theme` attribute
- Security level: `'loose'`

## 6. Media Handling

### 6.1 Data Flow

```
walkthrough.json               render_html.py                     Template
  media[].source_ref  ------>  resolve_media() matches to     --> data_uri, thumbnail_uri populated
  media[].data_uri    ------>  if exists, generate thumbnail  --> thumbnail_uri added
                               compress_image(): JPEG, 1280px max, <=200KB
                               make_thumbnail(): JPEG, 300px wide, quality=70
```

### 6.2 In-Slide Gallery

Rendered only if `step.evidence.media` exists (Jinja2 guard). The gallery container is:
```html
<div class="media-gallery" data-step-id="{{ step.id }}">
    <h4>Screenshots</h4>
    <div class="media-thumbs" id="media-{{ step.id }}"></div>
</div>
```

The `media-thumbs` div is **empty in the HTML** — populated at runtime by `initMediaGalleries()` from `DATA`.

### 6.3 Thumbnail Rendering

Each thumbnail is rendered by `renderThumb(item)`:
```html
<div class="media-thumb"
     data-full-src="{item.data_uri}"
     data-caption="{item.caption}">
  <img src="{item.thumbnail_uri || item.data_uri}" alt="{item.caption}" loading="lazy">
  <div class="media-caption">{item.caption}</div>   <!-- only if caption exists -->
</div>
```

### 6.4 Before/After Comparisons

When media items share a `group` value and have `group_role` values of `"before"` and `"after"`:
```html
<div class="media-compare">           <!-- 2-column grid -->
  <div class="media-compare-side">
    <div class="media-compare-label">Before</div>
    <!-- thumbnail -->
  </div>
  <div class="media-compare-side">
    <div class="media-compare-label">After</div>
    <!-- thumbnail -->
  </div>
</div>
```

Collapses to single column on viewports <= 768px.

### 6.5 Lightbox Behavior

1. Click on `.media-thumb` → collect all sibling thumbs in same container
2. Build `lightboxItems[]` array of `{src, caption}` from `data-full-src` and `data-caption` attributes
3. Display full image, caption text, prev/next buttons (hidden if only 1 image)
4. Navigation wraps around (modular arithmetic)
5. Close via X button, click outside, or `Escape`

## 7. Diff Rendering

### 7.1 Preprocessing (`render_html.py`)

`highlight_diff_hunk(hunk)` adds an `html` field:

| Input State | Processing |
|-------------|-----------|
| `hunk.diff` exists | Highlight `hunk.diff` with `DiffLexer` → `hunk.html` |
| `hunk.before` and/or `hunk.after` exist (no `diff`) | Synthesize `"- line"` / `"+ line"` format, highlight with `DiffLexer` → `hunk.html` |
| Neither `diff` nor `before`/`after` | No `html` added, hunk passes through unchanged |

The Pygments `HtmlFormatter` is configured with `nowrap=False, cssclass="highlight", nobackground=True`.

### 7.2 Template Rendering (JS)

In `renderEvidence()`, for each diff hunk:

```javascript
if (hunk.html) {
    // Use pre-rendered Pygments HTML directly
    html += hunk.html;
} else {
    // Fallback: render before/after as colored lines
    html += '<pre>' + this.renderDiffLines(hunk.before, hunk.after) + '</pre>';
}
```

`renderDiffLines(before, after)`:
- Splits `before` by newlines → wraps each in `<span class="diff-line-del">- {line}</span>`
- Splits `after` by newlines → wraps each in `<span class="diff-line-add">+ {line}</span>`

### 7.3 Diff Block Structure (in modal)

```html
<div class="diff-block">
  <div class="diff-header">
    <span class="diff-file-path">{hunk.file}</span>
    <span class="diff-editor-links">
      <a class="editor-link" href="vscode://file/{abs_path}:{line}">VS Code</a>
      <a class="editor-link" href="cursor://file/{abs_path}:{line}">Cursor</a>
    </span>
  </div>
  <div class="diff-content">
    <!-- Either hunk.html (Pygments) or <pre> with colored spans -->
  </div>
</div>
```

## 8. Editor Links

### 8.1 Path Resolution

The `absPath(filePath)` JS function:
```javascript
absPath(filePath) {
    const root = this.getRepoRoot();  // DATA.meta.repo_root, trailing slashes stripped
    if (!filePath) return '';
    if (filePath.startsWith('/') || !root) return filePath;
    return root + '/' + filePath;
}
```

### 8.2 Link Formats

| Editor | URL Format |
|--------|-----------|
| VS Code | `vscode://file/{absolute_path}:{line}` |
| Cursor | `cursor://file/{absolute_path}:{line}` |

### 8.3 Where Editor Links Appear

| Location | Context | Source |
|----------|---------|--------|
| Overview key_files | `cursor://file/...` links on each file chip | Jinja2 template |
| Step footer files_changed | `cursor://file/...` links on each meta tag | Jinja2 template |
| Diff hunk headers | Both `vscode://` and `cursor://` links | JS `editorLinks()` |
| Modal files_changed section | `cursor://file/...` links | JS `renderEvidence()` |

Note: The Jinja2 template only generates `cursor://` links for key_files and files_changed tags. The JS `editorLinks()` function generates both `vscode://` and `cursor://` links for diff hunk headers.

### 8.4 Dependency on `meta.repo_root`

If `meta.repo_root` is absent or empty:
- Jinja2: file paths are used as-is (no prefix)
- JS: `getRepoRoot()` returns `''`, `absPath()` returns the path unchanged
- Editor links will only work if file paths in the data are already absolute

## 9. Embedded Data (`DATA` Global)

The entire processed `data` dict is serialized to JSON and embedded as:
```html
<script>const DATA = {{ data_json }};</script>
```

This `DATA` object is the authoritative source for all JS-driven rendering. The Jinja2 template handles the static skeleton (TOC, step headers, claims, media gallery containers), while JS uses `DATA` to populate:
- Evidence modal content (diffs, commands, decisions, errors, media, files_changed)
- Media gallery thumbnails (in-slide)
- Lightbox image sources

The `data_json` variable is the `json.dumps()` output of the same `data` dict passed to Jinja2, so all fields are available to both Jinja2 and JS.
